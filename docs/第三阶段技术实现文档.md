# 第三阶段技术实现文档

## 1. 概述

第三阶段完成了 event-security 项目的核心后端集成，实现了从告警接收到事件生成、关联分析的全链路数据流。第四阶段将 M17 自动聚合引擎从 Node.js 迁移到 Flink，实现流式聚合。主要工作包括：

- **基础设施层**：PostgreSQL、Elasticsearch 7.16.3、Kafka、Redis 全栈集成
- **M16 DSL 校验与编译**：YAML DSL 解析、变量提取、ES 查询编译
- **M17 自动聚合引擎（Flink）**：Flink 流式聚合 → 模型匹配 → victim 维度分组 → ES 写入
- **M18 异步关联搜索**：DSL 执行 → 图谱生成 → 实体提取
- **M19 导入导出**：模型 JSON 格式导入导出
- **M20 审计日志**：所有写操作自动记录
- **去 Mock**：日志搜索、evidence、impact 全部切换到 ES 查询

## 3. 模块详细设计

### 3.1 M16 DSL 校验与编译 (`server/services/dsl.js`)

**职责**：解析 YAML DSL，校验结构，编译为 ES 查询。

**核心函数**：

```javascript
validateDsl(dslString) → { valid, compiled, variables, errors }
extractVariables(condition) → string[]
compileDsl(parsed) → CompiledStep[]
```

**DSL 结构要求**：
```yaml
processList:
  - condition: "字段 = \"值\" and 字段2 = \"值2\""
    source: "告警" | "日志" | "alert" | "log"
    time: ["-1h", "+1h"]  # 相对时间窗口
    relationship: "关联描述"
    essential: true | false  # 可选
    processUnit:  # 可选，嵌套步骤
      processList: [...]
```

**字段映射**（中文 DSL 字段 → ES 字段）：
```javascript
{
  '主机IP': 'attacker.ip',
  '源地址': 'source_address',
  '目的地址': 'destination_address',
  '文件MD5': 'hash.md5',
  '攻击者': 'attacker',
  '受害者': 'victim',
  // ...
}
```

**条件解析**：
- `field = "value"` → `{ term: { field: value } }`
- `field != "value"` → `{ bool: { must_not: [{ term: { field: value } }] } }`
- `field like "pattern%"` → `{ wildcard: { field: { value: "pattern*", case_insensitive: true } } }`
- `A and B` → `{ bool: { must: [...] } }`
- `A or B` → `{ bool: { should: [...], minimum_should_match: 1 } }`

**编译输出示例**：
```json
[{
  "index": "alert-index",
  "query": {
    "bool": {
      "must": [{ "term": { "attacker.ip": "10.10.1.100" } }],
      "filter": [{ "range": { "started_at": { "gte": "now-1h", "lte": "now+1h" } } }]
    }
  },
  "relationship": "攻击者IP关联",
  "essential": true,
  "variables": ["主机IP"],
  "processUnit": null
}]
```

### 3.2 M17 自动聚合引擎（Flink）

**职责**：消费 Kafka 告警，匹配模型，按 victim 维度聚合，写入 ES。

**实现位置**：`nsm-event-engine` 项目

| 类 | 路径 | 职责 |
|----|------|------|
| `AggregatorFunction` | `func/AggregatorFunction.java` | KeyedProcessFunction，核心聚合逻辑 |
| `EngineAggregatorEntry` | `engine/EngineAggregatorEntry.java` | 独立 Flink Job 入口 |

**处理流程**：
```
1. 消费 nsm-original-alert (keyBy victim/dst_ip/src_ip)
2. normalizeFlinkAlert() — 字段标准化 (src_ip→attacker, alarm_name→title)
3. matchesEntryConditions() — 匹配 PG ice_rule 的 entryAlerts
4. KeyedState 维护 incidentId + alertCount
5. 同一 victim → 更新已有 incident；新 victim → 创建 incident
6. ES 写入 incident-index + alert-index
7. 输出到 nsm-aggregated-alert (含 incident_id + model_id)
```

**告警标准化映射**：
```java
{
  "src_ip"     → "attacker",
  "dst_ip"     → "victim",
  "src_port"   → "attacker_port",
  "dst_port"   → "victim_port",
  "alarm_name" → "title",
  "alarm_content" → "description",
  "original_alert_level" → "severity" (3→high, 4→critical)
}
```

**模型匹配逻辑**：
```java
matchesEntryConditions(alert, conditions) → boolean
```
- 从 PG `ice_rule.raw` 加载 entryAlerts JSON 数组
- 每个 condition 的 `name` 字段匹配告警的 `rule_id` 或 `rule_name`
- 支持 `title` 通配符（`%` → `.*`）
- 空 entryAlerts → 匹配所有告警（开放模型）

**State 管理（Flink KeyedState）**：
- `ValueState<String> incidentId` — 当前 victim 关联的 incident ID
- `ValueState<Long> alertCount` — 该 incident 已聚合的告警数
- 按 victim 维度 keyBy，同一 victim 的告警自动合并

**双 PG 数据源**：
- `dataSource`（nsm 库）：主数据
- `socDataSource`（psoc001db 库）：ice_rule 表查询

**启动参数**：
```bash
--kafka.original_alert.topic nsm-original-alert
--kafka.aggregated_alert.topic nsm-aggregated-alert
--kafka.aggregator.group_id nsm-aggregator-local
--soc.pg.url jdbc:postgresql://localhost:5432/psoc001db
--es.host 127.0.0.1 --es.port 9200
```

**与原 Node.js Aggregator 的对比**：
| 维度 | Node.js Aggregator | Flink AggregatorFunction |
|------|-------------------|------------------------|
| 聚合方式 | 轮询 ES 查 incident | Flink KeyedState 维护 |
| 并发 | 单进程 | Flink 并行度可调 |
| 容错 | 无 | RocksDB + Checkpoint |
| ES 写入 | @elastic/elasticsearch | HttpURLConnection 直连 |

### 3.3 M18 异步关联搜索 (`server/services/searcher.js`)

**职责**：消费 Flink Aggregator 输出的聚合告警，执行 DSL，生成图谱和实体。

**输入**：Kafka `nsm-aggregated-alert`（Flink AggregatorFunction 输出）

**处理流程**：
```
1. 消费 nsm-aggregated-alert (兼容 Flink 格式和 Node.js 格式)
2. buildVariables(task) — 构建变量映射
3. loadModelDsl(modelId) — 从 PG 加载 DSL (statement 字段)
4. replaceVariables(rawDsl, variables) — 替换 ${变量} 占位符
5. validateDsl(resolvedDsl) — 校验 + 编译
6. executeCompiledSteps(compiled, context) — 执行 ES 查询
7. generateGraph(relatedAlerts, incidentId) — 生成图谱
8. extractEntities(relatedAlerts) — 提取实体
9. updateIncident(incidentId, { relatedAlerts, graph, entities })
10. createEntities(entities, incidentId) — 写入 entity-index
11. updateTaskProgress(incidentId, 100, 0) — 更新 PG 任务状态
```

**变量映射**：
```javascript
buildVariables(task) → {
  '受害者': task.victim,
  '源地址': task.victim,  // 或 task.attacker
  '目的地址': task.victim,
  '主机IP': task.victim,
  '攻击者': task.attacker
}
```

**图谱生成**：
```javascript
generateGraph(relatedAlerts, incidentId) → { nodes[], edges[] }
```
- 根节点：`incident:{id}`
- 告警节点：`alert:{alertId}`
- IP 节点：`ip:{value}`
- 主机节点：`host:{hostname}`
- 账号节点：`account:{username}`
- 文件节点：`file:{md5}`

**实体提取**：
```javascript
extractEntities(relatedAlerts) → {
  hosts: string[],
  ips: string[],
  accounts: string[],
  files: string[]
}
```

## 5. 关键代码路径

### 5.1 服务启动 (`server/index.js`)
```javascript
async function start() {
  await initDb();           // PG 建表 + seed
  await initEs();           // ES 建索引
  await ensureTopics();     // Kafka 建 topic
  await startAggregator();  // M17: 启动聚合引擎
  await startSearcher();    // M18: 启动搜索器
  server.listen(port);      // HTTP 服务
}
```

### 5.2 模型存储字段说明
| PG 字段 | 用途 | 格式 |
|---------|------|------|
| `statement` | DSL YAML（供 searcher 执行） | YAML 字符串 |
| `raw` | entryAlerts JSON（供 aggregator 匹配） | JSON 数组 |

**重要**：searcher 读取 `statement` 字段（DSL YAML），不是 `raw` 字段（entryAlerts JSON）。

### 5.3 ES 查询封装 (`server/es.js`)
```javascript
searchIndex(index, body)        // 搜索
indexDocument(index, id, doc)   // 单条写入
bulkIndex(index, docs[])        // 批量写入
```

## 6. 已知限制与后续优化

### 6.1 当前限制
1. **图谱丰富度**：依赖 DSL 条件与真实数据的匹配，测试数据有限
2. **实体提取**：仅从 searcher 查询结果提取，未集成外部威胁情报
3. **任务并发**：searcher 串行处理，未做并行优化
4. **错误处理**：部分场景仅有 console.error，未做重试机制

### 6.2 后续优化方向
1. 接入真实 Flink 引擎数据验证完整链路
2. 图谱布局算法优化（力导向、层次布局）
3. 实体风险评分集成外部情报
4. 事件处置流程（确认、处置、关闭）
5. WebSocket 实时推送事件更新

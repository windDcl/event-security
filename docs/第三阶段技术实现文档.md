# 第三阶段技术实现文档

## 1. 概述

第三阶段完成了 event-security 项目的核心后端集成，实现了从告警接收到事件生成、关联分析的全链路数据流。主要工作包括：

- **基础设施层**：PostgreSQL、Elasticsearch 7.16.3、Kafka、Redis 全栈集成
- **M16 DSL 校验与编译**：YAML DSL 解析、变量提取、ES 查询编译
- **M17 自动聚合引擎**：Kafka 消费 → 模型匹配 → 事件创建
- **M18 异步关联搜索**：DSL 执行 → 图谱生成 → 实体提取
- **M19 导入导出**：模型 JSON 格式导入导出
- **M20 审计日志**：所有写操作自动记录
- **去 Mock**：日志搜索、evidence、impact 全部切换到 ES 查询

## 3. 模块详细设计

### 3.1 M16 DSL 校验与编译 (`server/services/dsl.js`)

**职责**：解析 YAML DSL，校验结构，编译为 ES 查询。

**核心函数**：

```javascript
validateDsl(dslString) → { valid, compiled, variables, errors }
extractVariables(condition) → string[]
compileDsl(parsed) → CompiledStep[]
```

**DSL 结构要求**：
```yaml
processList:
  - condition: "字段 = \"值\" and 字段2 = \"值2\""
    source: "告警" | "日志" | "alert" | "log"
    time: ["-1h", "+1h"]  # 相对时间窗口
    relationship: "关联描述"
    essential: true | false  # 可选
    processUnit:  # 可选，嵌套步骤
      processList: [...]
```

**字段映射**（中文 DSL 字段 → ES 字段）：
```javascript
{
  '主机IP': 'attacker.ip',
  '源地址': 'source_address',
  '目的地址': 'destination_address',
  '文件MD5': 'hash.md5',
  '攻击者': 'attacker',
  '受害者': 'victim',
  // ...
}
```

**条件解析**：
- `field = "value"` → `{ term: { field: value } }`
- `field != "value"` → `{ bool: { must_not: [{ term: { field: value } }] } }`
- `field like "pattern%"` → `{ wildcard: { field: { value: "pattern*", case_insensitive: true } } }`
- `A and B` → `{ bool: { must: [...] } }`
- `A or B` → `{ bool: { should: [...], minimum_should_match: 1 } }`

**编译输出示例**：
```json
[{
  "index": "alert-index",
  "query": {
    "bool": {
      "must": [{ "term": { "attacker.ip": "10.10.1.100" } }],
      "filter": [{ "range": { "started_at": { "gte": "now-1h", "lte": "now+1h" } } }]
    }
  },
  "relationship": "攻击者IP关联",
  "essential": true,
  "variables": ["主机IP"],
  "processUnit": null
}]
```

### 3.2 M17 自动聚合引擎 (`server/services/aggregator.js`)

**职责**：消费 Kafka 告警，匹配模型，创建/更新 ES 事件。

**启动流程**：
```
1. refreshModels() — 从 PG 加载活跃模型到内存缓存
2. createConsumer('nsm-original-alert', 'aggregator-group', handler)
3. 每 30s 刷新模型缓存
```

**模型匹配逻辑**：
```javascript
matchesEntryAlerts(alert, entryAlerts) → boolean
```
- 如果模型无 entryAlerts → 匹配所有告警（开放模型）
- 否则：entry 的每个字段（name/type/severity/title）必须匹配告警对应字段
- title 支持 `%` 通配符

**事件创建流程**：
```
1. findMatchingModels(alert) → model[]
2. findExistingIncident(model.id, alert.victim) → existingHit | null
3. 如果存在 → updateIncident(existingHit, alert)
   否则 → buildIncident(alert, model) → createIncident(doc)
4. indexAlert(alert, incidentId) — 原始告警写入 alert-index
5. sendIncidentTask(incident, alert, model) — 触发 searcher
```

**事件文档结构**：
```javascript
{
  id: "inc-xxxxxxxx",
  title: alert.title || model.name,
  severity: alert.severity,
  attack_result: alert.result,
  scene_type: model.sceneType,
  ice_rule_id: model.id,
  attacker_array: [{ ip: "10.10.1.100", name: "" }],
  victim_array: [{ ip: "192.168.1.50", name: "" }],
  priority: severityToPriority(severity),  // critical=4, high=3, medium=2, low=1
  related_alerts: [alert],
  update_time: ISO8601
}
```

**任务消息格式**：
```javascript
{
  type: "incident_search",
  incidentId: "inc-xxxxxxxx",
  iceRuleId: "model-uuid",
  alertId: "alert-001",
  victim: "192.168.1.50",
  attacker: "10.10.1.100",
  startTime: ISO8601,
  endTime: ISO8601
}
```

### 3.3 M18 异步关联搜索 (`server/services/searcher.js`)

**职责**：消费搜索任务，执行 DSL，生成图谱和实体。

**处理流程**：
```
1. buildVariables(task) — 构建变量映射
2. loadModelDsl(modelId) — 从 PG 加载 DSL (statement 字段)
3. replaceVariables(rawDsl, variables) — 替换 ${变量} 占位符
4. validateDsl(resolvedDsl) — 校验 + 编译
5. executeCompiledSteps(compiled, context) — 执行 ES 查询
6. generateGraph(relatedAlerts, incidentId) — 生成图谱
7. extractEntities(relatedAlerts) — 提取实体
8. updateIncident(incidentId, { relatedAlerts, graph, entities })
9. createEntities(entities, incidentId) — 写入 entity-index
10. updateTaskProgress(incidentId, 100, 0) — 更新 PG 任务状态
```

**变量映射**：
```javascript
buildVariables(task) → {
  '受害者': task.victim,
  '源地址': task.victim,  // 或 task.attacker
  '目的地址': task.victim,
  '主机IP': task.victim,
  '攻击者': task.attacker
}
```

**图谱生成**：
```javascript
generateGraph(relatedAlerts, incidentId) → { nodes[], edges[] }
```
- 根节点：`incident:{id}`
- 告警节点：`alert:{alertId}`
- IP 节点：`ip:{value}`
- 主机节点：`host:{hostname}`
- 账号节点：`account:{username}`
- 文件节点：`file:{md5}`

**实体提取**：
```javascript
extractEntities(relatedAlerts) → {
  hosts: string[],
  ips: string[],
  accounts: string[],
  files: string[]
}
```

## 5. 关键代码路径

### 5.1 服务启动 (`server/index.js`)
```javascript
async function start() {
  await initDb();           // PG 建表 + seed
  await initEs();           // ES 建索引
  await ensureTopics();     // Kafka 建 topic
  await startAggregator();  // M17: 启动聚合引擎
  await startSearcher();    // M18: 启动搜索器
  server.listen(port);      // HTTP 服务
}
```

### 5.2 模型存储字段说明
| PG 字段 | 用途 | 格式 |
|---------|------|------|
| `statement` | DSL YAML（供 searcher 执行） | YAML 字符串 |
| `raw` | entryAlerts JSON（供 aggregator 匹配） | JSON 数组 |

**重要**：searcher 读取 `statement` 字段（DSL YAML），不是 `raw` 字段（entryAlerts JSON）。

### 5.3 ES 查询封装 (`server/es.js`)
```javascript
searchIndex(index, body)        // 搜索
indexDocument(index, id, doc)   // 单条写入
bulkIndex(index, docs[])        // 批量写入
```

## 6. 已知限制与后续优化

### 6.1 当前限制
1. **图谱丰富度**：依赖 DSL 条件与真实数据的匹配，测试数据有限
2. **实体提取**：仅从 searcher 查询结果提取，未集成外部威胁情报
3. **任务并发**：searcher 串行处理，未做并行优化
4. **错误处理**：部分场景仅有 console.error，未做重试机制

### 6.2 后续优化方向
1. 接入真实 Flink 引擎数据验证完整链路
2. 图谱布局算法优化（力导向、层次布局）
3. 实体风险评分集成外部情报
4. 事件处置流程（确认、处置、关闭）
5. WebSocket 实时推送事件更新
